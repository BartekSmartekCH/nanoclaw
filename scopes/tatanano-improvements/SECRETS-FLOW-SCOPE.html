Zero-.env Secrets Architecture

macOS Keychain as the single source of truth for TataNano

Project: TataNano (NanoClaw on Mac Mini) • Date: March 2026 • Status: Scope & Design

Executive Summary

This scope replaces TataNano's static .env file with macOS Keychain as the single source of truth for all secrets. Bartek saves credentials in the macOS Passwords app (on Mac or iPhone), and TataNano picks them up automatically — no SSH, no file editing, no restarts.

The Vision: Save a new Telegram bot token in Passwords on your iPhone while at a coffee shop. By the time you take your next sip, TataNano has detected it, created the bot, and registered the group — all without touching the Mac Mini.

Current Architecture

How Secrets Flow Today

Understanding the current flow is critical to designing a clean replacement. NanoClaw has a well-designed credential proxy that already isolates secrets from containers — we just need to change where the proxy reads them from.

1
Static file: data/env/env contains all secrets (CLAUDE_CODE_OAUTH_TOKEN, TELEGRAM_BOT_TOKEN, TELEGRAM_BOT_POOL) in plaintext
↓
2
readEnvFile() in src/env.ts parses the .env file and returns requested key-value pairs (does NOT load into process.env)
↓
3
Credential Proxy (src/credential-proxy.ts) reads ANTHROPIC_API_KEY / CLAUDE_CODE_OAUTH_TOKEN via readEnvFile() and injects them into upstream API requests
↓
4
Containers get ANTHROPIC_BASE_URL=http://host:3001 pointing to the proxy. The .env file is shadow-mounted to /dev/null — containers never see real secrets

Files Involved

File Role Change Needed
src/env.ts Parses .env file, returns key-value pairs Replace with Keychain reads
src/config.ts Reads non-secret config (ASSISTANT_NAME, BOT_POOL) Source config from Keychain too
src/credential-proxy.ts Reads secrets, proxies API calls Read from Keychain instead of .env
src/channels/telegram.ts Reads TELEGRAM_BOT_TOKEN from env Read from Keychain
launchd/com.nanoclaw.plist launchd service definition No change (already has no secrets)
data/env/env Plaintext secrets file Eliminated entirely

macOS Keychain Integration

Why Keychain

  • Hardware-backed encryption — secrets are encrypted by the Secure Enclave (Apple Silicon T2/M-series), never stored in plaintext on disk
  • iCloud Keychain sync — changes made on iPhone/iPad sync to Mac within seconds to minutes
  • Passwords app (macOS 15+) — first-class GUI for managing entries on Mac, iPhone, iPad
  • CLI access via security — read/write Keychain items from the command line, perfect for NanoClaw
  • No daemon required — Keychain is always available, managed by macOS itself

Keychain Naming Convention

All TataNano secrets use a structured naming convention for the Keychain service name:

# Pattern: tatanano.{category}.{name} # Core secrets (replace current .env): tatanano.anthropic.oauth-token # CLAUDE_CODE_OAUTH_TOKEN tatanano.anthropic.api-key # ANTHROPIC_API_KEY (if using API key mode) tatanano.telegram.bot-token # TELEGRAM_BOT_TOKEN (primary bot) # Bot pool tokens: tatanano.telegram.pool-bot-1 # First pool bot token tatanano.telegram.pool-bot-2 # Second pool bot token tatanano.telegram.pool-bot-3 # Third pool bot token # Dynamic new bots (auto-detected): tatanano.telegram.bot.my-new-bot # New bot, auto-registered # Future services: tatanano.openai.api-key # OpenAI key tatanano.github.token # GitHub PAT tatanano.custom.anything # Any custom secret

Reading from Keychain via CLI

macOS provides the security command for Keychain access. NanoClaw will use this to read secrets at startup:

# Read a single secret: security find-generic-password \ -s "tatanano.telegram.bot-token" \ -a "tatanano" \ -w # List all tatanano entries (for discovery/watching): security dump-keychain | grep -A4 "tatanano\." # Add a new secret (or via Passwords app): security add-generic-password \ -s "tatanano.telegram.bot.my-bot" \ -a "tatanano" \ -w "BOT_TOKEN_HERE" \ -U # Update if exists
First-run authorization: The first time NanoClaw reads a Keychain item, macOS shows a dialog: "node wants to access keychain item X". Click "Always Allow" to grant permanent access. This is a one-time per-item action. When running under launchd, the user must be logged in for this dialog to appear. After the first approval, reads are silent.

iCloud Keychain Sync Latency

Scenario Typical Latency Notes
iPhone to Mac (same WiFi) 5–30 seconds Both devices on same network, fastest path
iPhone to Mac (different networks) 30 seconds–2 minutes Routes through iCloud servers
After device wake from sleep Up to 5 minutes Mac Mini is always-on, so this is iPhone-side only
Worst case (network issues) Up to 15 minutes Rare; iCloud retries automatically
Practical implication: Since Mac Mini is always-on and always-connected, the typical sync time for saving on iPhone is 5–30 seconds. The polling interval (60s default) means a new secret will be detected within ~90 seconds on average.

Architecture Design

A. New Module: src/keychain.ts

A single new module that wraps all Keychain interactions. This replaces src/env.ts as the secrets source while keeping the same interface:

// src/keychain.ts — Keychain-backed secret reader import { execFileSync } from 'child_process'; const KEYCHAIN_ACCOUNT = 'tatanano'; /** Read a single secret from macOS Keychain. Returns undefined if not found. */ export function readSecret(service: string): string | undefined { try { return execFileSync('security', [ 'find-generic-password', '-s', service, '-a', KEYCHAIN_ACCOUNT, '-w', ], { encoding: 'utf-8', timeout: 5000 }).trim(); } catch { return undefined; } } /** Read multiple secrets, returning a key-value map. */ export function readSecrets( mapping: Record<string, string> ): Record<string, string> { const result: Record<string, string> = {}; for (const [envKey, service] of Object.entries(mapping)) { const value = readSecret(service); if (value) result[envKey] = value; } return result; } /** Discover all Keychain entries matching a prefix (e.g. "tatanano.telegram.bot.") */ export function discoverSecrets( prefix: string ): Array<{ service: string; value: string }> { ... }

B. Zero-.env Startup Flow

At startup, NanoClaw reads all required secrets from Keychain instead of the .env file. The credential proxy and Telegram channel use the same readSecrets() function:

Startup Sequence (Before → After)

1
credential-proxy.ts calls readSecrets({ 'ANTHROPIC_API_KEY': 'tatanano.anthropic.api-key', 'CLAUDE_CODE_OAUTH_TOKEN': 'tatanano.anthropic.oauth-token' }) instead of readEnvFile()
↓
2
telegram.ts calls readSecret('tatanano.telegram.bot-token') for the primary bot, and discovers pool bots via discoverSecrets('tatanano.telegram.pool-bot-')
↓
3
config.ts reads non-secret config via readSecret('tatanano.config.assistant-name') or keeps reading from .env for non-sensitive config only
↓
4
No .env file needed. The data/env/env file is deleted. Container shadow-mount of .env to /dev/null is no longer necessary (but kept for defense-in-depth).

C. Auto-Detection of New Secrets (Watcher)

A new watcher module polls the Keychain for entries matching the tatanano. prefix and detects when new entries appear:

// src/keychain-watcher.ts — Polls for new Keychain entries export function startKeychainWatcher(opts: { onNewSecret: (service: string, value: string) => void; pollInterval?: number; // default: 60_000 (60s) }) { const knownServices = new Set<string>(); // Seed with current entries on startup for (const entry of discoverSecrets('tatanano.')) { knownServices.add(entry.service); } // Poll for new entries setInterval(() => { for (const entry of discoverSecrets('tatanano.')) { if (!knownServices.has(entry.service)) { knownServices.add(entry.service); opts.onNewSecret(entry.service, entry.value); } } }, opts.pollInterval ?? 60_000); }

D. Auto-Registration Handlers

When the watcher detects a new Keychain entry, it routes to the appropriate handler based on the service name category:

// In src/index.ts — startup startKeychainWatcher({ onNewSecret: (service, value) => { if (service.startsWith('tatanano.telegram.bot.')) { // New Telegram bot detected! const botName = service.split('.').pop(); handleNewTelegramBot(botName, value); } if (service.startsWith('tatanano.telegram.pool-bot-')) { // New pool bot — hot-add to the pool addToPool(value); } if (service === 'tatanano.anthropic.oauth-token') { // OAuth token rotated — update credential proxy refreshCredentialProxy(value); } }, });

New Bot Creation Flow

This is the flagship use case: Bartek creates a new Telegram bot from his iPhone and TataNano auto-deploys it without any manual intervention on the Mac Mini.

Step 1: Create Bot on Telegram
1
Open Telegram on iPhone → message @BotFather → /newbot → name it "MyNewBot" → receive token 123456:ABC-xyz...
Step 2: Save to Passwords App
2
Open Passwords app on iPhone → tap + → add new entry:
Title: tatanano.telegram.bot.my-new-bot
Password: 123456:ABC-xyz... (the bot token)
Username: tatanano
Step 3: iCloud Sync
3
iCloud Keychain syncs the new entry to Mac Mini. Typical latency: 5–30 seconds on the same network, up to 2 minutes across networks.
Step 4: TataNano Auto-Detection
4
Keychain watcher's next poll discovers tatanano.telegram.bot.my-new-bot as a new entry. Triggers handleNewTelegramBot("my-new-bot", token).
Step 5: Bot Goes Live
5
Handler creates a new grammY Bot instance with the token, starts polling, and sends a confirmation message to the main group:
"New Telegram bot 'my-new-bot' detected and started. Send a message in that bot's chat, then use /register_group to activate it."
Total time from BotFather to live bot: Under 2 minutes in typical conditions. Bartek never touches the Mac Mini.

What About Group Registration?

The auto-detection starts the bot and makes it listen for messages. However, NanoClaw's group registration (which chat to respond in, with what trigger) still requires a /register_group command. Two approaches:

  1. Simple (recommended): Bot starts listening. Bartek sends a message in the desired chat. Then tells Andy in the main group: "Register the my-new-bot chat." Andy calls register_group via IPC.
  2. Advanced (future): Extended Keychain naming convention encodes registration info directly, e.g., tatanano.telegram.bot.my-new-bot.jid=tg:-1001234. The watcher auto-registers without human intervention.

Security Considerations

Threat Model Comparison

Threat .env File (Current) Keychain (Proposed)
Disk theft / unauthorized access Plaintext readable by anyone with file access Encrypted by Secure Enclave, requires user login
Malicious process reads secrets Any process with file read permissions macOS prompts for per-app Keychain access
Accidental git commit Possible (requires .gitignore discipline) Impossible (secrets never on filesystem)
Backup exposure Included in Time Machine if not excluded Keychain backed up encrypted, requires device unlock
Container escape Mitigated (shadow mount to /dev/null) Mitigated (same shadow mount + no file to find)

Keychain Access Control

Key detail: The security CLI inherits the calling user's Keychain access. Since NanoClaw runs as Bartek's user via launchd, it can read entries that Bartek has authorized. No root access or special entitlements needed.
launchd caveat: When NanoClaw runs under launchd (not in a terminal session), the first Keychain access may require a one-time GUI authorization dialog. This means Bartek needs to be logged in to the Mac Mini's GUI session (even if via Screen Sharing) for the initial setup. After "Always Allow" is granted, subsequent reads are silent even when no GUI session is active.

Migration Plan

One-Time Migration Script

#!/bin/bash # scripts/migrate-to-keychain.sh # Reads current .env and saves each secret to macOS Keychain set -e ENV_FILE="data/env/env" # Map .env keys to Keychain service names declare -A MAPPING=( ["CLAUDE_CODE_OAUTH_TOKEN"]="tatanano.anthropic.oauth-token" ["ANTHROPIC_API_KEY"]="tatanano.anthropic.api-key" ["TELEGRAM_BOT_TOKEN"]="tatanano.telegram.bot-token" ) while IFS='=' read -r key value; do [[ -z "$key" || "$key" == \#* ]] && continue service="${MAPPING[$key]}" if [[ -n "$service" ]]; then security add-generic-password \ -s "$service" -a "tatanano" -w "$value" -U echo "Migrated $key -> $service" fi done < "$ENV_FILE" # Handle TELEGRAM_BOT_POOL (comma-separated) POOL=$(grep "^TELEGRAM_BOT_POOL=" "$ENV_FILE" | cut -d= -f2) IFS=',' read -ra TOKENS <<< "$POOL" for i in "${!TOKENS[@]}"; do security add-generic-password \ -s "tatanano.telegram.pool-bot-$((i+1))" \ -a "tatanano" -w "${TOKENS[$i]}" -U echo "Migrated pool bot $((i+1))" done echo "Migration complete. Verify with: security dump-keychain | grep tatanano"

Rollback Plan

Phase 1 includes fallback to .env. If Keychain reads fail, NanoClaw falls back to the .env file. This means you can deploy Phase 1, verify everything works, and only then delete the .env file. If anything goes wrong, recreate the .env and restart — instant rollback.

Open Questions

  1. Non-secret config: Should ASSISTANT_NAME and other non-sensitive config also move to Keychain, or stay in launchd plist / a small config file? Keychain is overkill for non-secrets but keeps everything in one place.
  2. Passwords app UX on iPhone: The Passwords app is designed for website credentials (URL + username + password). We're using it for generic key-value storage. Need to test the exact UX for adding entries with a service name like tatanano.telegram.bot.my-bot. It may be more natural to use the "Notes" field of a single "TataNano" entry, or a Shortcut that wraps the creation.
  3. Multiple Keychains: Should TataNano secrets go in the login keychain (default) or a dedicated "TataNano" keychain? A dedicated keychain would be cleaner but adds complexity.
  4. Polling vs. file-system events: Is there a way to get notified when iCloud Keychain syncs a new entry, instead of polling? The kSecEventNotification API exists in Swift but may not be easily accessible from Node.js. Polling at 60s is simple and sufficient.
