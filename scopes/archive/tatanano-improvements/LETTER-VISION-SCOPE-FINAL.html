Design Principles

Mirror the voice.ts pattern exactly. Voice transcription already solves this problem: download media on the host, process on the host, pass text to the container. Image processing follows the same path. The agent skill receives structured text — never raw image data.
  • Host-side processing only — Apple Vision and Ollama never exposed to containers
  • Graceful degradation — if tools unavailable, fall back to [Photo - OCR unavailable]
  • IPC stays lean — large data stored as files, containers receive path references
  • Flat module structure — src/image-processor.ts, not a directory
  • Privacy-first — no data leaves the Mac mini unless Claude is explicitly invoked

Architecture: How It Fits the Codebase

The voice.ts template

The existing voice pipeline is the exact blueprint:

Voice message → telegram.ts downloads .ogg
→ voice.ts transcribes on HOST (Whisper)
→ Agent receives: [voice]: extracted text
→ Agent processes the text content

Image processing follows the same pattern:

Photo/PDF → telegram.ts downloads image
→ image-processor.ts extracts on HOST (Apple Vision + Ollama)
→ Agent receives: [image]: {text, structure, metadata} or file path ref
→ /letter skill processes the structured content

Why not call host.docker.internal from the container?

While host.docker.internal is already used for the credential proxy (confirmed in container-runtime.ts:15), that is a mandatory, always-running service. Ollama is optional. Coupling the container to an optional host service would create silent failures and fragility. The pre-processing approach keeps the boundary clean.

telegram.ts Change (line 437)

Before:

this.bot.on('message:photo', (ctx) => storeNonText(ctx, '[Photo]'));

After (mirrors voice handler at line 439):

this.bot.on('message:photo', async (ctx) => {
  if (!IMAGE_PROCESSOR_AVAILABLE) {
    storeNonText(ctx, '[Photo]');  // Graceful degradation
    return;
  }
  try {
    // Download largest photo size to temp dir
    const photo = ctx.message.photo[ctx.message.photo.length - 1];
    const file  = await ctx.getFile();
    const url   = `https://api.telegram.org/file/bot${this.botToken}/${file.file_path}`;
    const imgPath = await downloadToTemp(url, photo.file_unique_id);

    // Process on host (Apple Vision + Ollama)
    const result = await processImage(imgPath);
    if (!result) {
      storeNonText(ctx, '[Photo - processing failed]');
      return;
    }

    // Write result file (keeps IPC lean for multi-page)
    const filePath = writeImageResultFile(photo.file_unique_id, result);

    // Pass to agent as structured reference
    const content  = formatImageContent(result, filePath);
    const caption  = ctx.message.caption || '';
    this.opts.onMessage(chatJid, {
      content: caption ? `${content}\n${caption}` : content,
      sender: senderName,
      timestamp: new Date().toISOString(),
    });
  } catch (err) {
    logger.error({ err }, 'Error processing image');
    storeNonText(ctx, '[Photo - processing error]');
  }
});

// Also handle documents (PDFs, uncompressed images)
this.bot.on('message:document', async (ctx) => {
  const mime = ctx.message.document?.mime_type || '';
  if (mime.startsWith('image/') || mime === 'application/pdf') {
    // Same flow as photo handler above
  } else {
    storeNonText(ctx, '[Document]');
  }
});

IPC Payload Strategy

A 3-page letter scanned at typical phone resolution is 8–20 MB of base64. This must not go inline in the IPC message.

Rule: Image data never travels through IPC as base64. Only file path references.
  • Extracted text + structured data stored in /workspace/ipc/files/letter-{id}.json
  • Agent receives: [letter]: /workspace/ipc/files/letter-abc123.json
  • Agent reads the file directly — container has /workspace/ipc mounted
  • Files expire after 24h (cleanup job or on next session start)

letter-{id}.json schema

{
  "processed_at": "2026-03-20T22:00:00Z",
  "pages": 3,
  "extracted_text": "Full OCR text from all pages...",

The /letter Skill (SKILL.md)

Lighter than originally planned — pre-processing happens on the host. The skill is pure orchestration.

When triggered (/letter), the skill:

  1. Looks back in conversation for [letter]: markers (recent messages)
  2. Reads the referenced JSON file(s) from /workspace/ipc/files/
  3. Presents the structured summary to the user
  4. Asks 1–3 targeted questions if a reply is needed
  5. Drafts reply in the original language + English translation
  6. Iterates on request; optionally outputs as PDF
UX guidance in SKILL.md: User should send all letter photos first, then type /letter. The skill can also accept /letter reply — decline politely to combine in one step.
