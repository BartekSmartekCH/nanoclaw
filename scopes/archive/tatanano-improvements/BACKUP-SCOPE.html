📌Executive Summary

TataNano is a powerful AI agent running in a Docker container on Bartek's Mac Mini, accessible exclusively via Telegram. Without proper backups, Bartek's work, custom skills, and configuration could be lost if the container crashes or the Mac Mini fails.

✓ What We're Solving:
  • Automatic daily backup of workspace files (/workspace/group/) to GitHub
  • Version control for custom skills (send-pdf, etc.) in a separate skills repo
  • Secure backup of secrets (API keys, tokens) separate from code
  • One-command manual backup: @Claude /backup now
  • Full disaster recovery procedure documented and tested

Timeline: Estimated 2-3 hours to implement. No downtime required. Bartek manages everything via Telegram.

⚠️What's at Risk

Three Critical Assets

1. Workspace Files (/workspace/group/)

User's active work: scopes, documents, notes, logs, preferences.

  • Risk if lost: Weeks/months of work gone forever
  • Current state: Only exists in container's writable layer
  • Backup frequency: Daily auto-backup + manual on-demand

2. Custom Skills (/home/node/.claude/skills/)

Custom Telegram skills like send-pdf that extend TataNano's capabilities.

  • Risk if lost: Custom functionality disappears after container rebuild
  • Current state: Only exists in container's writable layer
  • Backup frequency: Automatically cloned at container startup

3. Secrets (/workspace/project/data/env/env)

API keys, bot tokens, OAuth credentials. These are critical but must NEVER go to GitHub.

  • Risk if lost: TataNano can't authenticate with Telegram, Anthropic, etc.
  • Risk if leaked: Anyone can impersonate Bartek's bot or steal his API quota
  • Current state: Only stored locally in the container
  • Backup approach: Separate secure backup (NOT in GitHub)
🚨 Critical Risk: If /workspace/project/data/env/env is committed to GitHub, attackers gain:
  • TELEGRAM_BOT_TOKEN — can operate Bartek's bot as an attacker
  • CLAUDE_CODE_OAUTH_TOKEN — can use Bartek's Anthropic API quota
  • Any other API keys stored in env file
Our solution: .gitignore + pre-commit scanning + GitHub push protection ensures secrets never reach GitHub. Secrets are backed up manually in iCloud Keychain.

🎯Backup Strategy

Three-Repo Architecture

Repository Contents Update Frequency Risk Level
nanoclaw-workspace (Private GitHub) /workspace/group/* (user's work) Daily + manual on-demand Low (no secrets)
tatanano-skills (Private GitHub) /home/node/.claude/skills/* (custom skills) Cloned at container startup; auto-committed when skills change Low (no secrets)
iCloud Keychain (Apple Passwords) /workspace/project/data/env/env (secrets only) Manual — save to Keychain whenever secrets change Critical (end-to-end encrypted by Apple)

Why Separate Repositories?

  • Security isolation: Secrets are never mixed with code
  • Access control: Workspace repo can be less restricted; secrets vault stays locked down
  • Recovery speed: Restore workspace files without touching secrets
  • Audit trail: Each repo has its own history for different retention policies

Backup Flow

1. User runs: @Claude /backup now (or scheduled daily at 2:00 AM UTC) 2. Backup skill: - cd /workspace/group - git add -A - git commit -m "Auto-backup: YYYY-MM-DD HH:MM:SS UTC" - git push -u origin main 3. Telegram notification: ✅ Backup complete 📦 Files: 427 ⏰ Timestamp: 2026-03-20 14:30:00 UTC 🔐 Secrets: NOT backed up (safe) 4. Logs: - Entry recorded in /workspace/group/logs/backup.log - GitHub shows commit history
✓ Key Safety Feature: The backup skill uses a .gitignore file that explicitly excludes /workspace/project/data/env/env, preventing accidental secret commits. This is enforced at the git level — even if Bartek runs git add -A manually, secrets won't be staged.

🛠️Implementation Steps

Step 1: Create GitHub Repositories (5 minutes)

  1. Go to GitHub.com and create two new private repositories:
    • nanoclaw-workspace — for /workspace/group/ files
    • tatanano-skills — for custom skills
  2. Clone the nanoclaw-workspace repo locally (on your development machine, not in the container)
  3. Create a .gitignore file in the repo:
    # Secrets — NEVER backup these *.env env .env* *.key *.pem *.secret *token* # Logs *.log # OS files .DS_Store .AppleDouble .LSOverride
  4. Commit and push the .gitignore

Step 2: Generate GitHub Personal Access Tokens (10 minutes)

  1. Go to GitHub.com → Settings → Developer settings → Personal access tokens
  2. Click "Generate new token" and create two tokens:
    • workspace-token — scoped to nanoclaw-workspace repo only
      • Permissions: repo (full control of private repositories)
      • Expiration: 90 days (rotate before expiration)
    • skills-token — scoped to tatanano-skills repo only
      • Permissions: repo
      • Expiration: 90 days
  3. Save both tokens securely (you'll only see them once)

Step 3: Configure TataNano Container (15 minutes)

  1. Update /workspace/project/data/env/env with GitHub tokens:
    GITHUB_USERNAME=BartekSmartekCH GITHUB_WORKSPACE_TOKEN=ghp_xxxxxxxxxxxxxxxxxxxx GITHUB_SKILLS_TOKEN=ghp_xxxxxxxxxxxxxxxxxxxx
  2. Update the Dockerfile to clone the skills repo at container startup:
    RUN git clone https://${GITHUB_USERNAME}:${GITHUB_SKILLS_TOKEN}@github.com/BartekSmartekCH/tatanano-skills.git /home/node/.claude/skills
  3. Rebuild the Docker image:
    cd [nanoclaw folder] docker build -f container/Dockerfile -t nanoclaw-agent . docker restart tatanano

Step 4: Install the /backup Skill (5 minutes)

  1. Copy the backup skill SKILL.md to the container:
    [nanoclaw folder]/container/skills/backup/SKILL.md
  2. Rebuild Docker image (as in Step 3)
  3. Register the /backup command in Telegram:
    • Go to @BotFather → /setcommands
    • Add: /backup — Backup workspace to GitHub

Step 5: Set Up Scheduled Auto-Backup (10 minutes)

  1. Create a scheduled task via Telegram:
    @Claude /schedule backup daily 2:00 AM
  2. TataNano will confirm:
    ✅ Scheduled: Daily auto-backup at 2:00 AM UTC Next run: 2026-03-21 02:00:00 UTC
  3. Task runs automatically every 24 hours; Bartek receives Telegram notification

Step 6: Save Secrets to iCloud Keychain (5 minutes)

Secrets are backed up manually using Apple's built-in password manager. No AWS account, no terminal commands, no extra apps needed.

  1. On your Mac, open System Settings → Passwords (or the Passwords app on macOS Sequoia+)
  2. Click + to add a new entry
  3. Title: "TataNano Secrets"
  4. In the Notes field, paste the full contents of /workspace/project/data/env/env (all 4 secrets)
  5. Save. Done — it syncs automatically to your iPhone and iPad via iCloud.
Why iCloud Keychain? It's end-to-end encrypted, requires Face ID / Touch ID, syncs across your Apple devices, and you already have it. No new accounts, no monthly fees, no terminal commands.

When to update: Any time you rotate a token or add a new secret, update the Keychain entry. TataNano sends a weekly reminder via Telegram.

Step 7: Test the Backup (5 minutes)

  1. Trigger manual backup:
    @Claude /backup now
  2. Check Telegram notification for success message
  3. Verify files appear in GitHub:
    • https://github.com/BartekSmartekCH/nanoclaw-workspace
    • https://github.com/BartekSmartekCH/tatanano-skills
  4. Check that secrets were NOT committed (open the GitHub repo and search for "TELEGRAM_BOT_TOKEN" — should find nothing)
💡 Total Setup Time: ~1.5 hours including testing. No downtime required — Bartek can continue using TataNano throughout the setup.

🔄Recovery Procedures

Scenario 1: Workspace Files Lost (Minor Disaster)

What happened: User accidentally deleted important files from /workspace/group/

Recovery time: 5 minutes

  1. Clone the workspace backup:
    git clone https://github.com/BartekSmartekCH/nanoclaw-workspace.git /tmp/workspace-restore
  2. Restore files to /workspace/group/:
    cp -r /tmp/workspace-restore/* /workspace/group/
  3. Verify files are restored and continue working

Scenario 2: Custom Skills Lost (Minor Disaster)

What happened: send-pdf skill disappeared after container rebuild

Recovery time: 2 minutes (automatic)

  1. Container automatically pulls skills repo at startup
  2. Skills are restored to /home/node/.claude/skills/
  3. No manual action needed

Scenario 3: Container Completely Lost (Major Disaster)

What happened: Mac Mini crashed and Docker container was deleted

Recovery time: 30 minutes

  1. Restore the Docker image:
    cd [nanoclaw folder] docker build -f container/Dockerfile -t nanoclaw-agent . docker run -it nanoclaw-agent
  2. Restore workspace files:
    git clone https://github.com/BartekSmartekCH/nanoclaw-workspace.git /tmp/restore cp -r /tmp/restore/* /workspace/group/
  3. Restore custom skills:
    git clone https://github.com/BartekSmartekCH/tatanano-skills.git /home/node/.claude/skills
  4. Restore secrets from iCloud Keychain:
    • On your iPhone or Mac, open Passwords (or Settings → Passwords)
    • Search for "TataNano Secrets"
    • Copy each secret value and paste into /workspace/project/data/env/env
    • All 4 secrets are also regenerable: Anthropic key from console.anthropic.com, Telegram tokens from @BotFather
  5. Verify recovery:
    @Claude /status
  6. Confirm all workspace files, skills, and environment variables are present

Scenario 4: Secrets Leaked (Security Incident)

What happened: GitHub token was accidentally committed or security breach detected

Recovery time: 15 minutes

  1. Immediately revoke leaked tokens:
    • Go to GitHub.com → Settings → Developer settings → Personal access tokens
    • Delete the compromised token (workspace-token or skills-token)
  2. Rotate Telegram bot token:
    • Go to @BotFather → /revoke (for the affected bot)
    • Create a new bot and get new TELEGRAM_BOT_TOKEN
    • Update /workspace/project/data/env/env in the container
    • Update the "TataNano Secrets" entry in iCloud Keychain
  3. Regenerate GitHub token:
    • Create new Fine-Grained PAT scoped to the backup repo only (contents:write + metadata:read)
    • Update GITHUB_TOKEN in /workspace/project/data/env/env
    • Update iCloud Keychain entry
    • Restart the container
    • Check the backup repo's git log for any unauthorized commits
  4. Optional: Clean GitHub history (if secrets were committed)
    git filter-branch --index-filter 'git rm --cached --force LEAKING_FILE' -- --all git push origin --force --all
  5. Monitor for unauthorized activity on Telegram bot and API usage
🔐 Prevention (4 layers): (1) .gitignore excludes secret-like files. (2) Pre-commit scan catches secrets pasted into documents. (3) GitHub Push Protection blocks known secret patterns server-side. (4) Regular token rotation (every 90 days) limits the window of exposure.

🔒Security Considerations

Defense-in-Depth: Multiple Layers

Layer Defense Mechanism Protects Against
Git .gitignore Explicitly excludes /workspace/project/data/env/env Accidental secret commits
Private GitHub Repos Only authenticated users can access repos Public disclosure of workspace or skills
Pre-Commit Scan Before each backup, scan staged files for secret patterns (sk-ant-, AAF, AAG, ghp_) Secrets that bypass .gitignore (e.g., pasted into a document)
GitHub Push Protection GitHub's built-in secret scanning blocks pushes containing known secret formats Server-side last resort if client-side checks fail
Fine-Grained PAT GitHub token scoped to only the backup repo with contents:write permission Lateral movement if token is compromised — attacker can only access one repo
iCloud Keychain Secrets backed up in Apple's end-to-end encrypted password manager Secrets loss if Mac Mini dies — recoverable from iPhone or other Mac
Token Expiration 90-day rotation policy for GitHub PAT Long-lived compromised credentials

GitHub Repository Security

  1. Branch Protection Rules:
    • Enable for the main branch in both repos
    • Require at least 1 review before merging (optional, since Bartek is the only user)
    • Enable "Require branches to be up to date before merging"
  2. Repository Settings:
    • Set both repos to Private
    • Disable "Allow force pushes" to prevent accidental history rewrites
    • Enable "Automatically delete head branches" to clean up after merges
  3. Secret Scanning & Push Protection:
    • Enable GitHub's built-in secret scanning: Settings → Code security → Secret scanning → Enable
    • Enable Push protection (same page) — this blocks pushes containing recognized secret patterns server-side
    • This is your last line of defense if .gitignore and pre-commit scanning both fail

Secrets Management Best Practices

✓ DO:
  • Save all secrets to iCloud Keychain (Apple Passwords) as a Secure Note
  • Update the Keychain entry every time you rotate a token
  • Rotate the GitHub PAT every 90 days
  • Use a Fine-Grained PAT scoped to only the backup repo
  • Immediately revoke any exposed tokens
  • Monitor GitHub notifications for suspicious activity
  • Check your Anthropic billing dashboard if you suspect API key compromise
✗ DON'T:
  • Commit secrets to GitHub (even in private repos, even encrypted)
  • Share secrets via email, Telegram, or any unencrypted channel
  • Use a Classic GitHub token — always use Fine-Grained PAT
  • Leave token expiration unset (always set to 90 days max)
  • Give the GitHub PAT more permissions than contents:write on the backup repo
  • Skip the .gitignore setup — it's your first line of defense
  • Use git push --force — it can overwrite backup history

Risk Assessment by Secret Type

TELEGRAM_BOT_TOKEN

  • If exposed: Attacker can send/receive messages from Bartek's bot, potentially impersonating him or accessing sensitive conversations
  • Mitigation: Revoke via @BotFather and create new token (5 minutes)
  • Rotation frequency: Every 90 days (or immediately if exposed)

CLAUDE_CODE_OAUTH_TOKEN

  • If exposed: Attacker gains access to Bartek's Claude Code credits and can use his API quota
  • Mitigation: Revoke token in Anthropic dashboard and generate new one
  • Rotation frequency: Every 90 days
  • Cost of exposure: High (attacker can consume expensive API quota)

TELEGRAM_BOT_POOL (3 worker bot tokens)

  • If exposed: Attacker can control 3 additional bots — wider blast radius than the main bot
  • Mitigation: Revoke all 3 via @BotFather and create new tokens
  • Rotation frequency: Every 90 days (or immediately if exposed)

GITHUB_TOKEN (backup PAT)

  • If exposed: Attacker can push malicious code to the backup repo or read its contents
  • Mitigation: Delete the PAT in GitHub Settings, check repo for unauthorized commits, generate new one
  • Rotation frequency: Every 90 days
  • Blast radius: Limited — Fine-Grained PAT only has access to the backup repo

📅Maintenance & Rotation

Weekly Checklist (5 minutes)

  • Check Telegram for backup notifications
    • Confirm daily auto-backup ran successfully
    • If you see errors, investigate immediately
  • Spot-check GitHub repos
    • Visit https://github.com/BartekSmartekCH/nanoclaw-workspace
    • Verify recent commits are there

Monthly Checklist (15 minutes)

  • Test manual backup:
    @Claude /backup now
  • Verify no secrets in GitHub:
    • Search both repos for "TELEGRAM_BOT_TOKEN", "TOKEN=", "API_KEY"
    • Should find nothing
  • Review GitHub commit history:
    • Ensure all commits are from Bartek (or the backup automation)
    • Watch for suspicious commits

Quarterly Checklist (30 minutes)

  • Token Rotation (CRITICAL):
    • GitHub PAT: Go to github.com → Settings → Developer Settings → Fine-grained tokens. Delete old, create new with same permissions (contents:write on backup repo only). Update env file and iCloud Keychain.
    • Consider rotating Telegram and Anthropic tokens if they haven't been rotated in 6+ months
  • Test disaster recovery:
    • Manually clone the backup repo and verify files are intact
    • Verify iCloud Keychain entry is up to date — open Passwords on your iPhone and check
  • Update documentation if procedures have changed

Backup Failure Response

Scenario: Backup notification shows error instead of success

  1. Check the error message in the Telegram notification
  2. Common issues and fixes:
    • "Authentication failed" → GitHub token expired or revoked
      • Fix: Generate new token, update container env
    • "Nothing to commit, working tree clean" → No files changed since last backup
      • Fix: This is normal — skill reports success anyway
    • "Network error" → Mac Mini lost internet connection
      • Fix: Reconnect to network, manually trigger backup again
    • "Permission denied" → Token doesn't have repo access
      • Fix: Regenerate token with correct scope in GitHub
  3. If issue persists, check /workspace/group/logs/backup.log for details

Annual Review

  • Assess backup strategy against new risks or changes in TataNano
  • Review GitHub repo settings and update branch protection rules if needed
  • Audit backup log to ensure consistent daily backups
  • Plan for any infrastructure changes (e.g., migrating Mac Mini, changing hosting)
